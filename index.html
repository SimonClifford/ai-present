What is AI Dementia?

The algorithm:

  1. Zoe’s group is in the Adaptive Brain Lab, Cambridge.
  2. Takes a 3D brain scan (a magnetic resonance image, MRI).
  3. Processes it to reduce it to only showing grey matter.
  4. Obtains some cognitive scores, usually from a database.
  5. Runs the fairly simply Python code to reduce all this to a single score.
  6. Uses a Generalised Metric Learning Vector Quantization, GMLVQ.

The method1,2 developed by Zoe Kourtzi’s group uses a machine learning (ML) algorithm trained on brain images (MRIs) and cognitive scores to give a prognostic score of the likelihood of the patient having Alzheimer’s.

1: https://doi.org/10.1016/j.nicl.2020.102199

2: https://doi.org/10.1101/2020.08.15.252601

Images

  1. We need for a user to be able to query a PACS for patient info, scan details, and finally download data.
  2. The prototype works on NIfTI images. A format commonly used to store MRI brain imaging data, popular with researchers, less so with clinicians.
  3. The segmentation uses standard SPM12 functions to leave only grey matter.

  • Use several tools to convert to NIfTI2 format.
  • Use existing SPM12 container to segment into grey matter image.

Building upon the prototype

  1. Auth could later be SSO or whatever.
  2. Search is within the PACS.
  3. Relevant studies are MRI scans, perhaps with a particular description.
  4. The download may be slow, or delayed.
  5. The score retrieval may also be delayed.
  6. The segmentation takes minutes at 100% CPU.
  7. The result is stored locally because the user may have wandered off!

The PIs want to take this from the prototype to something that can be used in a clinical setting.

  1. The user, a clinician, will authenticate to the app.
  2. They will then search for a patient, using patient ID, date of birth, and name.
  3. Once they find them, they choose from any relevant studies.
  4. When they select the study they want the request begins:
    1. The study is downloaded.
    2. The cognitive scores are retrieved.
    3. The study is converted to NIfTI, then segmented.
    4. The ML code is run on this and the cognitive scores.
    5. The result is stored locally.

SQLAlchemy1

  1. SQLAlchemy is a Python SQL toolkit and Object Relational Manager.
  2. Captures the description of each table in the DB in Python code.
  3. Only store the schema in one place.
  4. Can automatically create the entire DB. Extensions to migrate between versions.
  5. Provides Pythonic way to address database joins.
  6. So in example shown, each Request references a single User by its id (line 15).
  7. By declaring a relationship (line 16) we can reference the User object that the request is owned by directly. We also use back_populates to get a list of Requests that each User owns. All accessing the database is lazy.
class User(UserMixin, db.Model):
    
    id = db.Column(db.Integer, primary_key=True)
    username = db.Column(db.String(length=64), unique=True, index=True,
                         nullable=False)
    # as per https://www.rfc-editor.org/errata_search.php?rfc=3696&eid=1690
    email = db.Column(db.String(length=254))                     
    password_hash = db.Column(db.String(length=128), nullable=False)
    is_admin = db.Column(db.Boolean, nullable=False, default=False)
    requests = relationship('Request')

class Request(db.Model):

    id = db.Column(db.Integer, primary_key=True)  # internal ID
    user_id = db.Column(db.Integer, db.ForeignKey('user.id'), nullable=False)
    user = relationship('User', back_populates='requests')
    series_instance_uid = db.Column(db.String(length=64),
                                    nullable=False)  # UI
    patient_id = db.Column(db.Text, ForeignKey('patient.patient_id'),
                           nullable=False)  # LO
    patient = relationship('Patient')
    study_description = db.Column(db.Text)  # LO
    study_date = db.Column(db.Date)  # DA
    series_description = db.Column(db.Text)  # LO
    series_date = db.Column(db.Date)  # DA
    series_description = db.Column(db.Text)  # LO
    mr_scan_path = db.Column(db.Text)
    mr_scan_path_nifti = db.Column(db.Text)
    gm_scan_path = db.Column(db.Text)
    # will fix but currently scores parsed as
    # comma sep list by scripts
    score_data = db.Column(db.Text)
    # result_data stores JSON encoded result data.
    result_data = db.Column(db.Text, default='{}')
    request_date = db.Column(db.DateTime, server_default=func.now())
    result_date = db.Column(db.DateTime)
    # Choose 40 chars because we might use git commit SHAs as version
    # markers one day?
    result_version = db.Column(db.String(40))
    task_id = db.Column(db.String(36))

The segmentation process

  1. The segmentation process may be slow. Might take minutes, should probably run serially with other segmentations.
  2. But web requests are pretty much immediate (apart from streaming, etc).
  3. So we need an asynchronous system: user submits request, goes away, comes back later.
  4. Celery is a task queue sytem: put tasks in the queue, workers will process them.
  5. Celery needs a broker, effectively a fast database / cache, which manages the tasks.
  6. Redis can do this.
  7. Do we assume it’s there…?
  • Segmentation is quite slow.
  • A web request is expected to finish quickly.
  • Need asynchronous task management.
  • Celery1!.
    • But celery needs a broker.
